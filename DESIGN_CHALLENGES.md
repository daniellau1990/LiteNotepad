# LiteEdit 设计挑战与解决方案

## 1. 显示行号
**挑战**: 在 CodeMirror 6 编辑器中实现高性能行号显示，确保滚动流畅，即使在大文件中也是如此。

**解决方案**:
- 使用 CodeMirror 6 内置的 `lineNumbers()` 扩展，该扩展采用虚拟化渲染，仅渲染可见区域的行号。
- 通过 `EditorView.lineWrapping` 支持自动换行，同时保持行号准确。
- 行号区域使用绝对定位和 CSS 优化，避免重排开销。

**性能考虑**:
- 行号渲染在 GPU 加速的图层上完成，与文本内容分离。
- 监听滚动事件时使用防抖处理，避免频繁更新。

## 2. 性能约束: <1秒启动, <30MB 内存
**挑战**: Tauri 应用包含 Rust 后端和 React 前端，两者都需要在资源限制内快速启动。

### 启动时间优化策略:
- **代码分割**: 使用 Vite 的动态导入将 CodeMirror 语言支持按需加载。
  ```typescript
  // 动态导入语言模块
  const loadLanguage = async (ext: string) => {
    switch (ext) {
      case 'js': return import('@codemirror/lang-javascript')
      case 'md': return import('@codemirror/lang-markdown')
      case 'json': return import('@codemirror/lang-json')
      default: return null
    }
  }
  ```
- **Rust 优化**: 
  - 启用 LTO (Link Time Optimization) 和代码生成单元优化。
  - 使用 `strip = true` 移除调试符号。
  - 设置 `opt-level = 3` 进行最大程度优化。
- **前端优化**:
  - 使用 Vite 的 Rollup 分块，将 CodeMirror 核心与语言支持分离。
  - 启用 CSS 压缩和 Tree Shaking。

### 内存预算管理:
- **编辑器状态限制**:
  - 撤销历史记录限制为 100 步。
  - 大文件模式下禁用语法高亮和复杂扩展。
- **Rust 内存管理**:
  - 使用 `Box<>` 智能指针避免不必要的复制。
  - 文件读取使用流式处理，避免完整加载到内存。
- **React 优化**:
  - 使用 `React.memo` 避免不必要的组件重渲染。
  - 状态更新使用防抖和节流。

## 3. 文件系统安全
**挑战**: 仅允许通过用户对话框或系统文件关联打开的文件路径，防止任意文件访问。

**解决方案**:
- **Tauri allowlist 配置**:
  ```json
  {
    "fs": {
      "readFile": true,
      "writeFile": true,
      "readDir": false,  // 禁止目录遍历
      "copyFile": false,
      "createDir": false,
      "removeDir": false,
      "removeFile": false,
      "renameFile": false,
      "exists": false,
      "metadata": true
    }
  }
  ```
- **路径验证**: 所有文件操作前验证路径是否来自可信源（打开对话框或文件关联）。
- **沙箱模式**: 在开发环境中启用严格 CSP (Content Security Policy)。

## 4. 自动保存: 60秒定时 + 500ms 去抖 + 关闭事件保存
**挑战**: 实现可靠的无冲突自动保存机制，防止数据丢失。

### 状态机设计:
```
┌─────────┐   修改    ┌──────────┐   500ms无操作   ┌──────────┐
│  干净   │─────────▶│  脏状态  │───────────────▶│ 待保存   │
└─────────┘          └──────────┘                └──────────┘
     │                      │                          │
     │       保存完成       │       定时器60秒         │
     └──────────────────────┘───────────────▶──────────┘
```

### 实现策略:
- **三层保存触发**:
  1. **防抖保存**: 用户停止输入 500ms 后标记为脏状态。
  2. **定时保存**: 每 60 秒自动保存脏文件。
  3. **关闭保存**: 监听 `tauri://close-requested` 事件，阻塞关闭直到保存完成。
- **冲突处理**: 保存前检查文件修改时间，如果外部已修改，提示用户选择“覆盖”或“重新加载”。
- **错误恢复**: 保存失败时保留副本，提供恢复选项。

## 5. 大文件处理: >5MB 时支持语法高亮，启用流式读取
**挑战**: 大文件加载和编辑的性能问题，同时保持语法高亮功能。

### 阈值检测:
```rust
// Rust 后端提供文件大小检测
#[tauri::command]
fn get_file_size(path: String) -> Result<u64, String> {
    fs::metadata(&path).map(|m| m.len()).map_err(|e| e.to_string())
}
```

### 流式读取策略:
- **分块加载**: 文件 >5MB 时，按 1MB 块增量加载。
- **视图窗口**: CodeMirror 的 `Streaming` 扩展仅渲染可见区域。
- **增量语法分析**: 使用 CodeMirror 6 的增量解析器，仅对可见区域和附近文本进行语法高亮。

### 性能优化技术:
- **懒加载语法支持**: 语言解析器按需加载，大文件时使用 Web Worker 进行后台解析。
- **解析器节流**: 限制语法解析频率，用户停止输入 500ms 后开始解析。
- **内存映射优化**: 对于超大文件 (>50MB)，使用 `memmap2` 库进行内存映射读取，减少内存复制。
- **语法高亮缓存**: 已解析的语法高亮结果进行缓存，避免重复解析相同区域。

### 实现架构:
```
┌─────────────────┐   分块读取    ┌─────────────────┐
│   Rust 后端     │─────────────▶│   文件缓冲区    │
│  (内存映射/流式)│               │  (按需加载)     │
└─────────────────┘               └────────┬────────┘
                                           │ 增量更新
                                    ┌──────▼──────┐
                                    │ CodeMirror  │
                                    │   编辑器    │
                                    └──────┬──────┘
                                           │ 可见区域
                                    ┌──────▼──────┐
                                    │ 语法分析器  │
                                    │ (Web Worker)│
                                    └─────────────┘
```

## 6. 冲突检测: 外部修改文件时的处理策略
**挑战**: 检测文件在外部被修改的情况，防止数据覆盖。

### 检测机制:
- **轮询检查**: 每 30 秒检查打开文件的修改时间戳。
- **文件监视**: 使用 `notify` 库监听文件系统事件（Linux/macOS/Windows）。
- **内容哈希**: 保存文件内容的 SHA-256 哈希，用于检测静默修改。

### 用户交互流程:
```
检测到外部修改
    │
    ▼
┌──────────────┐
│  显示警告    │
│  文件已修改  │
└──────┬───────┘
       │
  ┌────┴─────┐
  ▼          ▼
覆盖       重新加载
本地更改   放弃本地更改
保留       加载新内容
```

### 实现细节:
- **三个状态**: `SYNCED`（同步）、`LOCAL_MODIFIED`（本地修改）、`EXTERNAL_MODIFIED`（外部修改）。
- **合并可能性**: 对于文本文件，提供简单的三向合并界面。
- **自动备份**: 冲突时自动创建 `.liteedit-backup` 文件。

## 性能监控与调优
- **启动时间测量**: 使用 `window.performance.now()` 记录各阶段加载时间。
- **内存使用监控**: 通过 `process.memoryUsage()`（Node.js）和 Rust 的 `#[global_allocator]` 跟踪内存分配。
- **分析工具**: 集成 `sentry` 用于错误报告，`vite-bundle-analyzer` 用于包大小分析。

## 跨平台注意事项
- **文件系统差异**: Windows 路径分隔符、macOS 沙箱限制、Linux 权限模型。
- **系统菜单集成**: 使用 Tauri 的 `tauri::menu` API 提供原生菜单体验。
- **文件关联注册**: 通过 Tauri 的 `tauri::updater` 配置各平台文件关联。

## 7. 打包与分发: 用户无需安装 Rust/Tauri CLI
**挑战**: 为用户提供开箱即用的体验，无需安装开发工具链。

### 解决方案:
- **预编译二进制分发**: 通过 CI/CD 管道为各平台构建独立安装包。
- **自动更新机制**: 集成 Tauri 自动更新功能，用户可一键升级。
- **最小化依赖**: 应用包含所有必要运行时，无需额外安装 Node.js 或 Rust。

### 打包策略:
- **Windows**: `.msi` 安装包 + `.exe` 便携版
- **macOS**: `.dmg` 磁盘映像 + `.app` 应用包
- **Linux**: `.AppImage` 通用包 + `.deb`/`.rpm` 系统包

### 构建自动化:
```json
// package.json 构建脚本
{
  "scripts": {
    "build:win": "tauri build --target x86_64-pc-windows-msvc",
    "build:mac": "tauri build --target aarch64-apple-darwin",
    "build:linux": "tauri build --target x86_64-unknown-linux-gnu",
    "package:all": "npm run build:win && npm run build:mac && npm run build:linux"
  }
}
```

### 用户安装流程:
```
用户访问发布页面
      │
      ▼
┌──────────────┐
│ 下载安装包   │
│ (自动检测平台)│
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 双击安装     │
│ (标准安装向导)│
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 立即使用     │
│ 无需配置环境 │
└──────────────┘
```

### 自动更新实现:
1. **版本检测**: 启动时检查 GitHub Releases 最新版本
2. **增量更新**: 下载差异包而非完整安装包
3. **静默安装**: 后台下载，用户确认后安装
4. **回滚机制**: 更新失败时自动恢复上一版本

### 开发与发布分离:
- **开发环境**: 需要 Node.js、Rust、Tauri CLI（仅开发者）
- **用户环境**: 仅需下载安装包，零配置使用
- **CI/CD 管道**: GitHub Actions 自动构建多平台包
