# LiteEdit TDD 开发规范

本文档定义 LiteEdit 项目的测试驱动开发（TDD）流程。所有新功能实现必须遵循此规范。

## TDD 核心循环

```
1. 红 (Red)    - 编写一个失败的测试
2. 绿 (Green)  - 编写最少代码使测试通过
3. 重构 (Refactor) - 改进代码结构，保持测试通过
4. 代码审查 (Review) - 确保代码质量符合标准
```

## 开发环境设置

### 测试框架
- **前端测试**: Vitest + React Testing Library
- **Rust 测试**: `cargo test` (内置测试框架)
- **E2E 测试**: Tauri 测试工具

### 运行测试命令
```bash
# 前端单元测试
npm test

# Rust 单元测试
cd src-tauri && cargo test

# 完整测试套件
npm run test:all
```

## 场景定义与任务列表

### 场景 1: 文件打开功能

#### 任务列表
1. **先写失败测试**
   - 创建测试文件 `tests/file_open.test.ts`
   - 编写测试：模拟用户点击"打开"按钮，验证文件选择对话框触发
   - 运行测试，确认失败

2. **验证失败**
   - 确保测试因功能未实现而失败
   - 记录失败信息作为基线

3. **实现**
   - 在 `src/components/MenuBar.tsx` 中实现 `handleOpenFile` 函数
   - 集成 Tauri `dialog.open` API
   - 确保代码最小化，仅使测试通过

4. **验证通过**
   - 重新运行测试，确认通过
   - 检查测试覆盖率是否达到要求

5. **重构**
   - 提取文件打开逻辑到独立的 Hook `useFileOpen`
   - 优化错误处理，添加用户提示
   - 确保代码符合项目规范

6. **代码审查**
   - 检查类型安全性（TypeScript strict mode）
   - 验证 Tauri allowlist 配置
   - 确保 UI 组件可访问性
   - 审核错误边界处理

## 测试质量要求

### 单元测试标准
- **覆盖率要求**: 核心逻辑 ≥ 90%
- **测试独立性**: 每个测试独立运行，不依赖外部状态
- **Mock 策略**: 外部依赖（文件系统、API）必须模拟
- **边界条件**: 必须测试异常路径和边界情况

### 集成测试标准
- **端到端流程**: 覆盖完整用户工作流
- **错误恢复**: 测试错误场景和恢复机制
- **性能基准**: 关键路径性能指标测试
- **跨平台验证**: 多平台行为一致性

### Rust 测试标准
- **内存安全**: 无内存泄漏、无数据竞争
- **错误处理**: 所有错误路径可测试
- **并发安全**: 多线程场景下的正确性
- **FFI 边界**: Tauri 命令接口的健壮性

## 重构指导原则

### 何时重构
1. 测试通过后立即重构
2. 发现代码重复时
3. 函数/方法过长时（> 20 行）
4. 模块职责不清晰时
5. 性能瓶颈识别后

### 重构规则
1. **不改变行为**: 重构前后测试结果必须一致
2. **小步前进**: 每次重构只解决一个问题
3. **持续测试**: 每次更改后运行测试套件
4. **版本控制**: 每个重构步骤单独提交

## 持续集成流程

### GitHub Actions 工作流
```yaml
name: TDD Pipeline
on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - 检查 TDD 规范符合性
      - 运行前端单元测试
      - 运行 Rust 单元测试
      - 生成测试覆盖率报告
      - 运行性能基准测试
      
  build:
    runs-on: ${{ matrix.os }}
    strategy:
      matrix:
        os: [windows-latest, macos-latest, ubuntu-latest]
    steps:
      - 构建跨平台应用包
      - 运行集成测试
      - 验证打包结果
```

### 质量门禁
1. **测试通过率**: 100% 必须通过
2. **覆盖率阈值**: 核心模块 ≥ 90%
3. **构建成功**: 所有平台构建成功
4. **性能基准**: 关键指标在阈值内
5. **安全检查**: 无已知安全漏洞

## 异常处理

### 测试失败处理流程
1. **立即停止**: 测试失败立即停止当前任务
2. **原因分析**: 确定失败原因（代码缺陷、测试错误、环境问题）
3. **修复验证**: 修复后重新运行完整测试套件
4. **文档更新**: 更新相关文档记录问题和解决方案

### 重构失败处理
1. **回滚**: 恢复到最后一个通过测试的版本
2. **分析**: 确定重构引入的问题
3. **分步实施**: 将重构拆解为更小的步骤
4. **重新测试**: 每步后验证测试通过

## 版本控制规范

### 版本号规则
遵循语义化版本（SemVer）：
- **主版本号**：不兼容的 API 修改
- **次版本号**：向下兼容的功能性新增
- **修订号**：向下兼容的问题修复

### 版本文件要求
1. **Version.md**：记录详细版本历史，每个版本必须包含：
   - 版本号（vX.Y.Z）
   - 发布日期（YYYY-MM-DD）
   - 新增功能列表
   - 修复问题列表
   - 技术改进说明
   - 已知问题说明

2. **package.json**：同步更新 version 字段
3. **src-tauri/Cargo.toml**：同步更新 version 字段

### Git 提交规范
1. **提交频率**：每个 TDD 任务完成后立即提交
2. **提交信息**：遵循 Conventional Commits 规范
3. **标签管理**：每个正式版本创建 git tag（vX.Y.Z）
4. **分支策略**：功能开发使用特性分支，主分支保持稳定
